# SuperPaste Bug Hunt

## Fixed issues

| Issue | Severity | Root cause | Fix | Regression test added |
| --- | --- | --- | --- | --- |
| Native document schema could strip newer slot/recipe fields during save or restart. | High | `src-tauri/src/engine.rs` had drifted behind the TypeScript models, so fields like `assemblyMode`, `steps`, and portable extra data were missing from the Rust-side structs. | Expanded the Rust structs to match the real document shape, added defaults for newer fields, and preserved unknown fields with `flatten` maps so newer content does not get erased when native persistence rewrites documents. | `src-tauri/src/engine.rs` -> `profiles_document_round_trips_recipe_steps_and_slot_assembly_mode` |
| Native save and panic-toggle paths could leave runtime state diverged from disk on persistence failure. | High | `save_clipboard_to_slot()` and `toggle_panic_mode()` mutated the live in-memory documents before `persist_documents()` succeeded. A failed write could leave the running app believing it had saved state that never reached disk. | Switched both paths to clone-first mutation helpers, persist the cloned documents, then swap the runtime state only after a successful write. | `src-tauri/src/engine.rs` -> `slot_save_prepares_new_documents_without_mutating_original`, `toggling_panic_mode_returns_a_new_document_snapshot` |
| Invalid config recovery was incomplete outside the native path. | High | The CLI store loader could fall back in memory, but it did not rewrite repaired documents back to disk or prefer `.bak` cleanly, which made restart behavior less trustworthy after corruption. | Added recovery-aware document loading in `src/cli/store.ts`, quarantine of invalid files, `.bak` preference, and self-healing rewrites of repaired or reseeded documents. | `src/cli/__tests__/store.test.ts` |
| `launch-superpaste.cmd` could fail immediately on healthy machines because it assumed `tauri dev` was the safe fallback path. | High | The launcher only checked for `npm`, then dropped into `npm run tauri dev` without repairing Cargo on `PATH`. In this Windows environment, `tauri dev` also hit an Application Control block on Rust debug build-script executables (`os error 4551`), so the "easy launcher" was fragile even when the toolchain was installed. | Moved the launcher logic into a testable Node script, added Cargo `PATH` repair from the default rustup install, switched the default launcher path to "build local release then launch", and kept `--dev` as an explicit opt-in for developer sessions. | `src/launcher/__tests__/launch-superpaste.test.ts` |
| Saving an empty clipboard could silently erase a slot. | Medium | Save-to-slot logic treated blank clipboard text as legitimate content. Under user error or timing races, a useful slot could be overwritten with whitespace. | Guarded both the TypeScript engine and the native save path so blank/whitespace clipboards become a no-op with explicit status feedback. | `src/core/__tests__/superpaste-engine.test.ts` -> `does not overwrite a slot when the clipboard is empty` |
| Hotkey conflicts and reserved bindings were easy to miss. | Medium | Conflict detection existed in the domain model, but warnings were not strongly surfaced in the product shell, and reserved/invalid native bindings could degrade registration quality without clear feedback. | Surfaced conflict warnings in the dock/status bar, upgraded the native hotkey refresh path to degrade per-binding instead of aborting wholesale, and marked reserved bindings explicitly. | `src/domain/__tests__/hotkeys.test.ts` |
| Capture hotkeys saved clips but did not load the combo stack. | Medium | The native save path persisted the slot and stopped, so the fast "copy several things, then discharge" workflow still required dock clicks. | Native save now emits a queue-slot renderer command when `experimental.autoQueueCaptures` is enabled, and the editor exposes a Save only / Queue switch. | `src/hooks/useSuperPasteApp.ts` |
| Terminal/editor copy chords were brittle when the active app did not match the detector. | Medium | Capture used exactly one copy chord per host class. If a target ignored that chord, the slot save failed even when the alternate chord would have worked. | Native capture now logs attempted chords and retries with the alternate `Ctrl+C` / `Ctrl+Shift+C` chord before leaving the slot unchanged. | `src-tauri/src/engine.rs` -> `terminal_windows_prefer_shift_modified_clipboard_chords` |
| Manual OS-level smoke testing had no controlled target window. | Medium | Real focused-app paste behavior is hard to automate in CI, and the app did not provide a stable local target for repeated save/paste/restore verification. | Added a lightweight test-harness window that opens from the main shell and provides an always-focusable textarea plus exact hotkey smoke instructions. | `src/components/__tests__/TestHarnessWindow.test.tsx` |
| CLI failure paths were under-tested. | Medium | Error reporting for bad imports, missing profile ids, and empty previews was present, but not locked by tests. | Added explicit CLI regression coverage for missing/unknown profiles, invalid packs, and preview-without-output failures. | `src/cli/__tests__/commands.test.ts` |

## Remaining caveats

| Issue | Severity | Root cause | Current handling | Follow-up |
| --- | --- | --- | --- | --- |
| Clipboard restore is still text-only. | Low | The native coordinator uses a text clipboard transaction instead of a full multi-format clipboard snapshot. | Paste is blocked when a text snapshot cannot be captured, and the status message now reflects text-clipboard behavior more honestly. | Upgrade the native clipboard transaction to preserve richer clipboard formats. |
| Native workspace-path matching is still weak outside title/process rules. | Low | The current Windows active-window sampler returns title, process name, and process path, but not a reliable workspace path for every host app. | Profile auto-switching works from title/process rules and manual override, and the caveat is documented in QA/manual smoke. | Add host-specific workspace extraction for editors/terminals that expose it reliably. |
| Remove-last is still dock-driven. | Low | Paste, clear, and replay have native hotkeys, but there is still no dedicated persisted binding for removing only the last queued move. | Kept remove-last on the dock until the hotkey schema grows another explicit command binding. | Add a native remove-last command if it proves worth one more global shortcut. |
